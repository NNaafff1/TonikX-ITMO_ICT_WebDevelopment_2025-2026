# Выводы

1. Выполненные работы
- Реализована модель предметной области (Owner, DriverLicense, Car, Ownership) средствами Django ORM, выполнены миграции и созданы тестовые данные.
- Разработан REST API на базе Django REST Framework с CRUD‑операциями для основных сущностей:
  - Owners: list, retrieve, create (с вложенной записью driver_license), update (PUT/PATCH), delete.
  - Cars: list/retrieve/create/update/delete.
  - Ownership: создание записи владения с валидацией временных интервалов.
- Реализована валидация бизнес‑правила: для одной пары (owner, car) периоды владения не пересекаются — проверка выполнена в модели Ownership (clean/save).
- Добавлена аутентификация через JWT (SimpleJWT) и интеграция с Djoser (регистрация/управление пользователями).
- Подготовлена OpenAPI‑схема и UI (Swagger / ReDoc) с помощью drf‑spectacular.
- Подготовлена документация проекта в формате MkDocs (страницы: модель данных, эндпойнты, инструкция запуска, скриншоты).

2. Результаты тестирования
- Скрипт заполнения (populate_lab3_practice3_1.py) успешно создал 6–7 владельцев, 5–6 автомобилей и связал владельцев с автомобилями (по 1–3 владения на владельца).
- API корректно возвращает вложенные объекты (owner → driver_license, owner → ownerships → car) в одном GET запросе.
- Попытки создать пересекающиеся периоды владения отклоняются с ошибкой валидации.

3. Достоинства решения
- Чёткая модель данных с явными FK/OneToOne/unique ограничениями и валидацией сложного бизнес‑правила.
- Вложенные сериализаторы обеспечивают удобный формат ответа для клиент‑части.
- Наличие OpenAPI документации облегчает тестирование и интеграцию.
- Простая схема авторизации (JWT + Djoser) готова к использованию в фронтенде.

4. Ограничения и возможные улучшения
- Отсутствует авторизация и разграничение прав доступа для отдельных операций (на текущем этапе допускается доступ без прав). Рекомендуется добавить permission_classes (IsAuthenticated, IsAdminUser, кастомные права).
- Валидация Ownership выполняется в модели; для большой нагрузки рекомендуется атомарные транзакции и/или оптимизированные проверки на уровне БД.
- Нет тестов (unit/integration). Для зачёта и для устойчивости стоит добавить Django тесты (models/tests.py, api/tests.py).
- Для продакшн‑развёртывания: настроить CORS, HTTPS, rate limiting, логирование запросов и мониторинг.
